Request for Comments (RFC): New Documentation Site​

Overview​

This RFC outlines the requirements and proposed implementation for a new documentation site aimed at providing a high-quality, user-friendly, and easily maintainable resource for our product offerings.

High-Level Requirements​

  1. High-Level Products: The documentation site should cover all high-level products comprehensively.
  2. Independent Pipeline for Development and Release: The site should have a separate pipeline for development and release, ensuring that updates can be made independently of platform releases.
  3. Not Tied to Platform Releases: The documentation updates should not be tied to platform releases, allowing for more flexibility and faster iteration.
  4. Navigation Pane: The site should support a navigation pane with nested, expandable, and collapsible items for better organization and user experience.
  5. Page Ordering: Pages should be easily orderable to maintain logical flow and structure.
  6. Snippet Embedding: The ability to embed code snippets and other reusable content should be supported.
  7. IDE: Integration with popular Integrated Development Environments (IDEs) for easier content creation and editing.
  8. Lightweight and Performant: The site should be lightweight and performant, ensuring fast load times and a smooth user experience.
  9. Easy Creation and Editing: Content creation and editing should be straightforward, using a widely adopted markup language such as Markdown.
  10. Diagrams: Support for diagrams, potentially using tools like Mermaid.
  11. Free to Use: The framework and tools used should be free to use.
  12. Community Support: The solution should have a large, active community for support and contributions.
  13. Hosting: The site should be hosted on AWS for reliability and scalability.

Examples for Reference​

  • Stitch Docs
  • Rapyd Docs
  • Revio Docs

Frameworks to Consider​

  • Docusaurus: An optimized site generator built in React, suitable for creating documentation websites, blogs, and marketing pages.

Proposed Solution​

After evaluating various frameworks and tools, Docusaurus stands out as the most suitable option for our new documentation site. Here’s why:

  1. Independent Pipeline: Docusaurus supports independent pipelines for development and release, facilitating continuous integration and deployment.
  2. Flexible Updates: Documentation updates are not tied to platform releases, allowing for more frequent and flexible content updates.
  3. Navigation Pane: Docusaurus supports a highly customizable navigation pane with nested, expandable, and collapsible items.
  4. Page Ordering: Pages can be easily ordered through configuration files.
  5. Snippet Embedding: Supports embedding code snippets, making it easier to include reusable content.
  6. IDE Integration: Compatible with popular IDEs, enhancing the content creation and editing experience.
  7. Performance: Docusaurus is lightweight and performant, ensuring fast load times and a smooth user experience.
  8. Markdown Support: Content can be written in Markdown, a widely used and easy-to-learn markup language.
  9. Diagram Support: Supports diagrams using tools like Mermaid.
  10. Cost: Docusaurus is free to use.
  11. Community Support: It has a large, active community providing free support and contributions.
  12. AWS Hosting: Can be easily hosted on AWS, ensuring reliability and scalability.

Implementation Plan​

  1. Setup Docusaurus: Initialize a new Docusaurus project and configure it according to our requirements.
  2. Content Migration: Migrate existing documentation to the new site, ensuring proper organization and structure.
  3. Customization: Customize the look and feel of the site to align with our branding.
  4. Testing: Thoroughly test the site for performance, usability, and accuracy.
  5. Deployment: Deploy the site on AWS, setting up necessary CI/CD pipelines for continuous integration and deployment.
  6. Training: Provide training and documentation for the team on how to use and maintain the new site.

IDE Integration and Contribution Workflow​

To ensure easy integration with popular IDEs like Visual Studio Code and streamline the contribution process, the following steps will be taken:

  1. VS Code Integration:

    • Markdown Extensions: Recommend extensions such as "Markdown All in One" and "Markdown Preview Enhanced" for an enhanced editing experience.
    • Live Preview: Utilize the built-in live preview feature in VS Code to allow contributors to see their changes in real-time.
    • Linting and Formatting: Configure Markdown linting and formatting tools to ensure consistency and quality in documentation.

  2. Contribution Workflow:

    • Repository Setup: Host the documentation site on a Git repository (e.g., GitHub) to facilitate version control and collaboration.
    • Contribution Guidelines: Create comprehensive contribution guidelines, including how to clone the repository, make changes, and submit pull requests.
    • Issue Tracking: Use GitHub Issues to track tasks, bugs, and feature requests related to the documentation.
    • Pull Requests: Encourage contributions through pull requests, with a review process to ensure quality and consistency.
    • CI/CD Integration: Set up continuous integration and deployment pipelines to automatically build and deploy the site upon merging pull requests.

Running the Site Locally​

To run the Docusaurus site locally before deploying it, follow these steps:

  1. Initialize Docusaurus: npx create-docusaurus@latest my-docs-site classic cd my-docs-site

  2. Install Dependencies: npm install

  3. Run the Development Server: npm run start This command will start a local development server. You can view the site in your browser at http://localhost:3000 .

  4. Customize and Add Content:

    • Customize the docusaurus.config.js file for branding and navigation.
    • Add initial documentation content in the docs folder using Markdown.
    • Implement sample diagrams using Mermaid in Markdown files.

  5. Test Locally:

    • Make changes and see them reflected in real-time using the live preview.
    • Ensure that all features, such as navigation, page ordering, snippet embedding, and diagrams, are working as expected.

  6. Share the Local Setup:

    • Share the local setup instructions with your team so they can also run the site locally and contribute.

Conclusion​

By adopting Docusaurus for our new documentation site and integrating it with popular IDEs like VS Code, we can ensure a high-quality, user-friendly, and easily maintainable resource for our product offerings. Running the site locally before deployment allows us to test and refine the documentation thoroughly. This approach aligns with our requirements for performance, flexibility, and community support, while also being cost-effective and scalable.

By following this RFC, we aim to create a robust and efficient documentation site that will serve our users and developers effectively.
